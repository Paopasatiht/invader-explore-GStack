## Problem Statement

Thai men default to "nice guy" behavior — emotionally available, conflict-avoidant, 
approval-seeking in relationships — and lose the women they want as a result. 
The Don Juan Return YouTube channel (100k+ followers, Thailand) has validated this 
problem publicly. Existing solutions are passive (YouTube, TikTok) or expensive 
(10,000 baht / ~$300 USD courses). There is no interactive, on-demand, affordable 
option between "watch for free" and "pay for a course."

## Demand Evidence

- Don Juan Return YouTube channel: ~100k followers in Thailand (self-selected, 
  high-intent audience on exactly this topic)
- Existing course pricing: 10,000 baht ($300 USD) — people pay real money for 
  this methodology
- Builder's own demo: 400 conversations already generated with first version
- Builder (Paopow, 25-35, AI engineer) is himself the target user
- All 4 current workarounds are active in Thailand: passive content, paid courses, 
  peer communities, solo trial-and-error

## Status Quo

Thai men cobble together: passive YouTube/TikTok consumption, occasional 10k-baht 
course purchases, LINE group peer advice, and solo trial-and-error. None are 
interactive, private, or on-demand. The gap between "free but passive" and 
"expensive but human" is where this product lives.

## Target User & Narrowest Wedge

**The user:** Paopow. Working professional, 25-35, self-aware enough to know he's 
"too nice" and wants to change. Has disposable income. Won't share this publicly 
but would use it privately every day.

**Narrowest wedge — one interaction, no setup required:**
User pastes a specific real situation with a specific woman (e.g., "she left me 
on read for 3 days, we've gone on 2 dates"). Bot gives:
1. One-sentence diagnosis of what happened (Don Juan framing)
2. One concrete next action
3. One thing not to do

That's the complete v1 user experience. No onboarding, no profile, no lesson plan.

**Product philosophy applied to UX:** The product is slightly hard to access. 
3 free conversations per day. Email required after conversation #1 to return. 
Exclusivity IS the Don Juan principle — the product lives the methodology.

## Bot Persona & Content Policy

**The Don Juan framework (operational definition):**
The Don Juan methodology teaches men to operate from a position of self-respect 
rather than approval-seeking. In every situation analysis, the bot applies these 
rules in order:
1. Is this behavior coming from neediness or strength? (Diagnose the frame)
2. What would a man who has options do here? (Generate the action)
3. What would a "nice guy" do that would make this worse? (Flag the trap)

**System prompt structure (implement exactly):**
```
You are Don Juan — a confident, wise relationship coach for Thai men. You help 
men operate from self-respect rather than neediness. You are direct and warm, 
never harsh or manipulative. You do not demean women.

For every situation a user shares, respond with exactly this structure:
DIAGNOSIS: [one sentence on whether the user is acting from neediness or strength]
ACTION: [one concrete next action — specific behavior, not vague advice]
TRAP: [one specific "nice guy" behavior to avoid in this situation]

Keep each section to 1-3 sentences. Never give more than 3 action options.
Never give general life advice. Stay focused on the specific situation described.

If the user asks about career, mental health, illegal behavior, or anything 
outside relationship dynamics, respond only with: 
"I'm built specifically for relationship situations. What's going on with her?"
```

**Tone:** Direct, confident, warm. Think: wise older brother who's been through it.

**Example input/output:**
- Input: "She texted me 'hey' after 4 days of silence. I really like her. Should I 
  reply right away?"
- Output:
  DIAGNOSIS: You're about to act from neediness — replying instantly signals you 
  were waiting.
  ACTION: Wait 3-6 hours, then reply: "Hey, what's up?" Short. Don't ask how 
  she's been. You're busy.
  TRAP: Replying in 2 minutes, or asking "where have you been?" — both confirm 
  she had power over your attention.

**Absolute content limits:**
- No advice on coercion, manipulation of intoxicated individuals, harassment, 
  or assault
- No claims the methodology works in all situations
- No demeaning women as a category

## Infrastructure (Demo Phase — 50 Users)

**Intentionally minimal stack. No over-engineering.**

**What to build:**
- Frontend: Next.js deployed to Vercel (free tier)
- AI: OpenAI GPT-4o-mini with system prompt (see Bot Persona section above) + 
  manually curated excerpts from the book (selected by hand, not bulk-scraped)
- Storage: localStorage for conversation history within a session. No database.
  Implication: conversation history is lost when user closes tab — acceptable 
  for demo. Cross-session memory is a Phase 2 feature requiring a DB.
- Email gate flow (implement exactly):
  1. User completes conversation 1 (defined as: user sends 1 message, bot replies)
  2. Bot response is displayed normally
  3. On conversation 2 attempt: show full-screen modal with copy: 
     "Continue your conversation — enter your email to unlock." 
     Email field + "Continue" button. No skip option.
  4. On submit: POST to a Make.com (Integromat) webhook → stores to Google Sheet 
     row: [timestamp, email, invite_code, conversation_count]. 
     Use Make.com (not raw Google Sheets API) — handles auth, rate limits, PDPA 
     data export if needed.
  5. On 200 response from webhook: unlock conversation 2 in localStorage. 
     If webhook fails: still unlock (don't block user for infra failure), 
     log error to console.
  6. Daily limit (3 conversations): tracked in localStorage as 
     {date: "YYYY-MM-DD", count: N}. Reset at midnight Bangkok time (UTC+7). 
     If localStorage cleared: limit resets. Acceptable for demo — trust users.
- Access control: 
  - Hardcode 3-5 invite codes in an env variable (e.g., INVITE_CODES=djr2026,paopow,pilot)
  - On page load: if URL param ?code= matches a valid code → set localStorage flag 
    and allow access. If no code → show waitlist page ("Join the waitlist").
  - Codes do not expire in v1. No per-user code tracking needed for 50 users.
- Instrumentation: No admin dashboard. Use OpenAI usage dashboard 
  (platform.openai.com) to track total API calls and token usage. 
  For conversation completion metric (Week 3 gate): add a single event log line 
  to the Make.com webhook on each conversation completion — 
  {email, turns_completed, date}. Count rows in Google Sheet manually.

**What NOT to build for the demo:**
- User accounts or login system
- Persistent cross-session conversation history (drops email-summary premium feature 
  from v1 scope — premium email summary requires server-side history, implement in v2)
- Payment integration
- Thai language if translation slows build — English v1, Thai v2
- Admin dashboard

**PDPA note (Thailand Personal Data Protection Act):**
Collecting Thai user emails requires: (1) a consent checkbox on the email gate modal 
stating what the data is used for, (2) a one-sentence privacy statement. 
Minimum copy: "Your email is stored securely and used only to save your progress. 
We will not sell or share it." Add this before launch, even for 50-user demo.

**Cost model:**
- GPT-4o-mini pricing: ~$0.00015/1K input tokens, ~$0.0006/1K output tokens
- 50 users × 3 turns/day × 800 avg tokens/turn × $0.00015 = ~$0.018/day = ~$0.54/month
- Well within $50/month budget. GPT-4o-mini is the confirmed model for demo.
- If quality is insufficient after internal testing: use GPT-4o for first turn only 
  (hook quality), GPT-4o-mini for subsequent turns. Recalculate cost before switching.
